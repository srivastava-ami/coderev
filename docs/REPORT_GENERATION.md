# HTML Report Generation & JavaScript Safety

## Overview

The coderev HTML report is generated by Go code that embeds JavaScript into an HTML template. This document explains:
1. How JavaScript is embedded safely
2. Why comprehensive validation is necessary
3. The validation mechanisms in place
4. Guidelines for future developers

## Architecture

### Generation Pipeline

```
Report data (Go struct)
    ↓
Generator (internal/report/generator.go)
    ├→ Embed JavaScript files (//go:embed)
    ├→ Marshal report as JSON
    ├→ Execute template (html/template)
    ├→ Inject report JSON (template.JS type — safe)
    └→ Inject concatenated JS (template.JS type — safe)
    ↓
Generated HTML file (.coderev/report.html)
```

### JavaScript Files Embedded

Located in `internal/report/static/`:
- `report-core.js` — Boot, header, warnings, summary, violations, pillars
- `report-misc.js` — Files, architecture, docs, exceptions, AI review
- `report-arch.js` — Architecture visualization, package map
- `report-graph.js` — Interactive code graph
- `report-viz.js` — Utilities, HTML escaping, markdown parsing
- `report-utils.js` — Additional helpers

Total: ~2000 lines of JavaScript across 6 files.

## Safety Mechanisms

### 1. Type-Safe Template Rendering

**File:** `internal/report/generator.go`

```go
// Report JSON is injected as template.JS (safe)
ReportJSON: template.JS(reportJSON)

// JavaScript content is injected as template.JS (safe)
JS: template.JS(jsContent)
```

The `template.JS` type ensures data is HTML-escaped properly.

### 2. Data Escaping

**File:** `internal/report/static/report-viz.js`

All user-controlled data (file paths, rule IDs, messages, etc.) is escaped via `escHtml()` before rendering:

```javascript
function escHtml(s) {
  if (!s) return '';
  return String(s)
    .replace(/&/g,'&amp;')
    .replace(/</g,'&lt;')
    .replace(/>/g,'&gt;')
    .replace(/"/g,'&quot;');
}
```

### 3. Pre-commit Hook Validation

**File:** `.git/hooks/pre-commit`

Validates known broken patterns BEFORE commits:

```bash
# Check for known broken patterns (=> {) is the most common error)
grep -q "=> {)" "$js_file"
```

**Note:** The JavaScript files are fragments (not standalone) that get concatenated by Go. Comprehensive validation happens in the unit test (TestGenerateHTMLReportValid) which validates the complete generated HTML.

**Why this two-layer approach?**
1. **Pattern check** (pre-commit) — catches known issues immediately
2. **Unit test** (go test) — validates the complete generated HTML with all JavaScript together

### 4. Unit Test Validation

**File:** `internal/report/template_test.go`

Test: `TestGenerateHTMLReportValid()` validates:
- Report generation succeeds
- Essential functions present (renderHTML, renderWarnings, etc.)
- Proper arrow function syntax in `.map()` calls
- Proper closure of `.renderHTML()` calls with `)).join(''))`
- No broken syntax patterns (`=> {)`, unclosed parens, etc.)

Run with: `go test ./internal/report`

## Common JavaScript Syntax Errors (What We Protect Against)

### Error 1: Broken Arrow Function Closure
```javascript
// ❌ WRONG: Closing paren in wrong place
renderHTML(grid, pillars.map(p => {)
  const pct = Math.round(p.Score * 100);
  return `<div>...</div>`;
}).join('');

// ✅ CORRECT: Proper arrow function syntax
renderHTML(grid, pillars.map(p => {
  const pct = Math.round(p.Score * 100);
  return `<div>...</div>`;
}).join(''));  // Extra ) closes renderHTML() call
```

### Error 2: Missing Closing Parenthesis
```javascript
// ❌ WRONG: renderHTML call not closed
renderHTML(body, filtered.map((f, i) => {
  return `<tr>...</tr>`;
}).join('');  // Missing ) to close renderHTML()

// ✅ CORRECT: Both .map() and renderHTML() properly closed
renderHTML(body, filtered.map((f, i) => {
  return `<tr>...</tr>`;
}).join(''));  // Added closing ) for renderHTML()
```

### Error 3: Unclosed Template Literal
```javascript
// ❌ WRONG: Template literal not closed
const html = `<div>${name}`;  // Missing closing backtick

// ✅ CORRECT: Template literal closed
const html = `<div>${name}</div>`;
```

## Guidelines for Developers

### When Adding or Modifying JavaScript in Reports:

1. **Edit the `.js` file** in `internal/report/static/` (not the Go code)
   - Easier to test — just edit and reload in browser
   - Changes automatically picked up by `//go:embed`

2. **Test locally** before committing:
   ```bash
   ./coderev .
   # Open .coderev/report.html in browser
   # Check browser console for errors
   ```

3. **Pre-commit hook will validate**:
   - `git commit` will fail if JavaScript has syntax errors
   - Fix the error and try again

4. **Unit test will catch** if you try to commit invalid HTML:
   - `go test ./internal/report` validates generation
   - Checks for known broken patterns

5. **If you add NEW JavaScript patterns**:
   - Update the test with regex pattern validation
   - Update this document with the pattern explanation
   - Ensure pre-commit hook can catch it (or add to Node.js validation)

### DO NOT:

- Generate JavaScript strings dynamically in Go code
- Concatenate unvalidated JavaScript in templates
- Skip the pre-commit hook validation
- Modify JavaScript without testing in browser first

## Troubleshooting

**Q: Pre-commit hook blocked my commit with "JavaScript syntax error"**

A: Fix the syntax error:
1. Check the file mentioned in the error
2. Run `node -c internal/report/static/<filename>.js` to see the error
3. Fix the error (usually missing bracket/paren/quote)
4. Try committing again

**Q: Report generates but shows "no data" in browser**

A: Check browser console for JavaScript errors:
1. Open `.coderev/report.html` in browser
2. Open Developer Tools (F12)
3. Check the Console tab
4. The error message will show the line with the problem
5. Fix the JavaScript file and regenerate

**Q: I added JavaScript but Node.js isn't available**

A: The pre-commit hook falls back to regex pattern matching. However, for comprehensive validation, install Node.js:
```bash
# macOS
brew install node

# Ubuntu/Debian
sudo apt-get install nodejs

# Or download from https://nodejs.org/
```
